# Vault-Audit

A security middleware prototype that sits in front of MongoDB, intercepts every query, classifies it as normal or malicious, and writes tamper-proof audit logs.

Built as an academic prototype demonstrating how a lightweight ML layer can augment traditional rule-based access control on a database.

---

## Architecture

```
Caller (HTTP)
     │
     ▼
 api.py  ──────────────────  FastAPI layer
     │                       Routes: /query  /health  /audit/logs  /audit/stats
     ▼
 middleware.py  ────────────  Intercept layer
     │                       Feature extraction → threat scoring → audit log → block or pass
     ▼
 svm_engine.py  ────────────  Phase 3: Calibrated SVM classifier
     │                       Falls back to rule-based scorer if model not loaded
     ▼
 MongoDB (localhost:27017)
     ├── sensitive_data      10 mock employee records
     └── audit_logs          Tamper-stamped log of every query
```

## Setup

**Prerequisites:** Python 3.14, MongoDB running on localhost:27017.

```bash
# Clone and enter the project
git clone https://github.com/vpadival/vault-audit.git
cd vault-audit

# Create and activate virtual environment
python -m venv .venv
.venv\Scripts\activate        # Windows
# source .venv/bin/activate   # macOS / Linux

# Install dependencies
pip install fastapi uvicorn pymongo scikit-learn joblib faker numpy requests pytest
```

## Running

### 1. Initialise the database

Run once to create collections, indexes, and populate 10 mock employee records:

```bash
python db_setup.py
```

### 2. Train the SVM model

Run once (or any time you want to retrain):

```bash
python train_svm.py
```

Output: `svm_model.joblib` and `training_report.txt` in the project root.

### 3. Start the API server

```bash
python api.py
```

The server starts at `http://127.0.0.1:8000`. On startup you should see:

```
[svm_engine] Model loaded from ...\vault-audit\svm_model.joblib
```

## API Routes

| Method | Route | Description |
|---|---|---|
| GET | `/health` | Liveness probe — returns service and SVM status |
| POST | `/query` | Submit a query — intercept, score, log, return or block |
| GET | `/audit/logs` | Recent audit entries (`?flagged_only=true` supported) |
| GET | `/audit/stats` | Total / flagged / blocked counts + avg threat score |

### Example: clean query

```bash
curl -X POST http://127.0.0.1:8000/query \
  -H "Content-Type: application/json" \
  -d "{\"query_type\": \"READ\", \"query_filter\": {\"employee_id\": \"EMP-1001\"}, \"user_id\": \"alice\"}"
```

### Example: blocked query

```bash
curl -X POST http://127.0.0.1:8000/query \
  -H "Content-Type: application/json" \
  -d "{\"query_type\": \"DELETE\", \"query_filter\": {}, \"user_id\": \"attacker\"}"
```

Returns HTTP 403 with `"status": "blocked"`.

## Threat Scoring

Every query is scored on a 0.0–1.0 scale using a 6-feature vector:

| Feature | Description |
|---|---|
| `record_count` | Number of documents matched by the filter |
| `is_empty_filter` | 1 if no filter supplied (full collection scan) |
| `has_sensitive` | 1 if filter references ssn, bank_account, or salary |
| `is_high_risk_op` | 1 if operation is DELETE |
| `exceeds_limit` | 1 if record_count > 3 |
| `bulk_sensitive` | 1 if empty filter AND sensitive fields present |

**Thresholds:**

| Score | Gate | Action |
|---|---|---|
| < 0.50 | clean | Query executes normally |
| ≥ 0.50 | flagged | Query executes, logged with `flagged: true` |
| ≥ 0.85 | blocked | Query rejected, HTTP 403 returned |

**Phase 2** used a hand-tuned rule-based scorer. **Phase 3** replaced it with a calibrated RBF-SVM (98.7% CV accuracy, 100% recall on the blocked class). The rule-based scorer remains as an automatic fallback if `svm_model.joblib` is not found.

---

## SVM Model

- **Pipeline:** `StandardScaler → CalibratedClassifierCV(SVC, method=sigmoid, cv=5)`
- **Training data:** 5,400 samples generated from the rule-based scorer + hard-coded edge cases
- **CV accuracy:** 0.9870 ± 0.0009
- **Blocked class F1:** 1.0000 (zero false negatives)

The model file `svm_model.joblib` is excluded from git. Regenerate it with `python train_svm.py`.

## Database Schema

**`sensitive_data`** — 10 mock employees (Faker-generated, no real PII):

```
employee_id, name, ssn, email, department, role, salary, bank_account
```

**`audit_logs`** — one entry per query:

```
timestamp, user_id, query_type, record_count, threat_score,
integrity_hash, query_filter, ip_address, flagged
```

Indexes on `timestamp`, `user_id`, `threat_score` (audit_logs) and unique `employee_id` (sensitive_data).

## Constraints

This is an academic prototype. It is not production-hardened:

- No authentication on MongoDB
- Local MongoDB only — no Atlas or cloud
- Fake PII only — all records are Faker-generated
- Single machine — no distributed setup
- Fixed dataset — 10 employee records in `sensitive_data`
