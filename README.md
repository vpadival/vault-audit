# Vault-Audit

A security middleware prototype that sits in front of MongoDB, intercepts every query, classifies it as normal or malicious, and writes tamper-proof audit logs.

Built as an academic prototype demonstrating how a lightweight ML layer can augment traditional rule-based access control on a database.

---

## Architecture

```
Caller (HTTP)
     │
     ▼
 api.py  ──────────────────  FastAPI layer
     │                       Routes: /query  /health  /dashboard  /audit/logs  /audit/stats  /audit/verify
     ▼
 middleware.py  ────────────  Intercept layer
     │                       Feature extraction → threat scoring → chained audit log → block or pass
     ▼
 svm_engine.py  ────────────  Phase 3: Calibrated SVM classifier
     │                       Falls back to rule-based scorer if model not loaded
     ▼
 MongoDB (localhost:27017)
     ├── sensitive_data      10 mock employee records
     └── audit_logs          SHA-256-chained, tamper-evident log of every query
```

## Deploying (Render)

This repo ships with a `render.yaml` blueprint, so it deploys as a single web service.

1. Push this repo to GitHub.
2. Create a free [MongoDB Atlas](https://www.mongodb.com/cloud/atlas/register) cluster (or use a Render-hosted Mongo instance backed by a persistent disk) and grab its connection string.
3. On Render: **New → Blueprint**, point it at this repo. Render will read `render.yaml` automatically.
4. In the service's Environment tab, set `MONGO_URI` to your Atlas/Mongo connection string (this is the one variable `render.yaml` leaves for you to fill in manually, since it's a secret).
5. Deploy. The build step installs dependencies and trains the SVM model (`train_svm.py`, which needs no DB); the start step seeds the database (`db_setup.py`) and launches the API (`api.py`), which now binds to `0.0.0.0:$PORT` in production.
6. Once live, the dashboard is served at `https://<your-service>.onrender.com/dashboard` — it calls the API same-origin, so no extra config is needed there.

Notes:
- **Vercel** isn't a good fit for this project: Vercel's Python support is serverless functions, which conflicts with this app's persistent Mongo connection and in-memory SVM model loaded at startup.
- **GitHub Pages** can't run this at all — it only serves static files, and this app is a live FastAPI + MongoDB service.

## Running

### 1. Initialise the database

Run once to create collections, indexes, and populate 10 mock employee records:

```bash
python db_setup.py
```

### 2. Train the SVM model

Run once (or any time you want to retrain):

```bash
python train_svm.py
```

Output: `svm_model.joblib` and `training_report.txt` in the project root.

### 3. Start the API server

```bash
python api.py
```

The server starts at `http://127.0.0.1:8000`. On startup you should see:

```
[svm_engine] Model loaded from ...\vault-audit\svm_model.joblib
```

### 4. Open the dashboard

Navigate to `http://127.0.0.1:8000/dashboard` in your browser for the interactive web UI.

> **Note:** To use a non-local MongoDB, set the `MONGO_URI` environment variable before starting:
> ```bash
> set MONGO_URI=mongodb://user:pass@host:27017/   # Windows
> export MONGO_URI=mongodb://user:pass@host:27017/ # macOS / Linux
> ```

## API Routes

| Method | Route | Description |
|---|---|---|
| GET | `/` | Root — links to docs and all endpoints |
| GET | `/health` | Liveness probe — returns service and SVM status |
| GET | `/dashboard` | Interactive web UI |
| POST | `/query` | Submit a query — intercept, score, log, return or block |
| GET | `/audit/logs` | Recent audit entries (`?limit=N&flagged_only=true`, max 500) |
| GET | `/audit/stats` | Total / flagged / blocked counts + avg threat score |
| GET | `/audit/verify` | Walk the audit log chain and verify integrity |

### Example: clean query

```bash
curl -X POST http://127.0.0.1:8000/query \
  -H "Content-Type: application/json" \
  -d "{\"query_type\": \"READ\", \"query_filter\": {\"employee_id\": \"EMP-1001\"}, \"user_id\": \"alice\"}"
```

### Example: flagged query (salary dump)

```bash
curl -X POST http://127.0.0.1:8000/query \
  -H "Content-Type: application/json" \
  -d "{\"query_type\": \"READ\", \"query_filter\": {\"salary\": {\"$gt\": 0}}, \"user_id\": \"eve\"}"
```

Returns HTTP 200 with `"flagged": true` and threat score ~0.55.

### Example: blocked query

```bash
curl -X POST http://127.0.0.1:8000/query \
  -H "Content-Type: application/json" \
  -d "{\"query_type\": \"DELETE\", \"query_filter\": {}, \"user_id\": \"attacker\"}"
```

Returns HTTP 403 with `"status": "blocked"`.

### Example: verify the audit log chain

```bash
curl http://127.0.0.1:8000/audit/verify
```

Returns `{"valid": true, "total": N, "first_break_seq": null, "reason": null}` on a clean chain.

## Threat Scoring

Every query is scored on a 0.0–1.0 scale using a 6-feature vector:

| Feature | Description | Weight (rule-based) |
|---|---|---|
| `record_count` | Number of documents matched by the filter | input to `exceeds_limit` |
| `is_empty_filter` | 1 if no filter supplied (full collection scan) | +0.45 |
| `has_sensitive` | 1 if filter references ssn, bank_account, or salary | +0.20 |
| `is_high_risk_op` | 1 if operation is DELETE | +0.30 |
| `exceeds_limit` | 1 if record_count > 3 | +0.35 |
| `bulk_sensitive` | 1 if empty filter AND sensitive fields present | +0.05 |

**Thresholds:**

| Score | Gate | Action |
|---|---|---|
| < 0.50 | clean | Query executes normally |
| >= 0.50 | flagged | Query executes, logged with `flagged: true` |
| >= 0.85 | blocked | Query rejected, HTTP 403 returned |

**Example scores:**

| Query | Score | Result |
|---|---|---|
| Single employee lookup | 0.00 | clean |
| Single SSN lookup | 0.20 | clean |
| Salary dump (all records) | 0.55 | flagged |
| Full collection scan READ | 0.80 | flagged |
| DELETE by department | 0.30 | clean |
| Bulk DELETE no filter | 1.00 | blocked |

**Phase 2** used a hand-tuned rule-based scorer. **Phase 3** replaced it with a calibrated RBF-SVM. The rule-based scorer remains as an automatic fallback if `svm_model.joblib` is not found — useful for CI or first-run before training.

## SVM Model

- **Pipeline:** `StandardScaler -> CalibratedClassifierCV(SVC, method=sigmoid, cv=5)`
- **Training data:** 5,400 samples generated from the rule-based scorer + hard-coded edge cases
- **CV accuracy:** 0.9870 +/- 0.0009
- **Blocked class F1:** 1.0000 (zero false negatives)

The model file `svm_model.joblib` is excluded from git. Regenerate with:

```bash
del svm_model.joblib        # Windows — remove stale model first
# rm svm_model.joblib       # macOS / Linux

python train_svm.py
python api.py
```

> **Important:** After changing any scoring weights in `middleware.py`, apply the same change to
> `train_svm.py`'s `_rule_score()` and retrain. The two must stay in sync or the SVM will learn
> a different decision surface than the fallback scorer enforces.

## Tamper-evident Audit Log (Phase 4)

Every entry in `audit_logs` is part of a SHA-256 hash chain. Each entry's `integrity_hash` is computed over its own fields **plus** the previous entry's hash and a monotonic `seq` number. Any retrospective edit, insert, or deletion anywhere in the log breaks every hash downstream from the tamper point.

**Chain fields added to every log entry:**

| Field | Description |
|---|---|
| `seq` | Monotonic counter, starts at 0, unique index in MongoDB |
| `prev_hash` | The previous entry's `integrity_hash`, or `"GENESIS"` for seq 0 |
| `integrity_hash` | SHA-256 over `{timestamp, user_id, query_type, record_count, threat_score, query_filter, prev_hash, seq}` |

**Verifier behaviour:**

| Failure mode | Detected as |
|---|---|
| Field modified after write | `integrity_hash mismatch at seq=N` |
| Entry deleted from the middle | `seq mismatch: expected N, found M` |
| `prev_hash` link broken | `prev_hash mismatch at seq=N` |

**CLI verifier** — for offline checks (no server required):

```bash
python verify_chain.py          # pass/fail, exits 0 (clean) or 1 (broken)
python verify_chain.py -v       # also prints last 5 entries with prev -> hash links
```

## Database Schema

**`sensitive_data`** — 10 mock employees (Faker-generated, no real PII):

```
employee_id, name, ssn, email, department, role, salary, bank_account
```

**`audit_logs`** — one entry per query, part of a SHA-256 hash chain:

```
seq, timestamp, user_id, query_type, record_count, threat_score,
integrity_hash, prev_hash, query_filter, ip_address, flagged
```

Indexes on `timestamp`, `user_id`, `threat_score`, and unique `seq` (audit_logs); unique `employee_id` (sensitive_data).

## Bug Fixes & Improvements

Changes applied on top of the original Phase 1–4 implementation:

| Area | Fix |
|---|---|
| **Scoring bug** | `bulk_sensitive` branch had an always-False guard (`and not is_empty_filter`) — removed |
| **Scoring weight** | `has_sensitive` weight raised `0.10 -> 0.20` so bulk sensitive reads score >= 0.50 (flagged) |
| **Training sync** | Same weight fix applied to `train_svm.py` `_rule_score()` so SVM trains on consistent labels |
| **Configuration** | MongoDB URI reads from `MONGO_URI` env-var (localhost fallback for dev) |
| **DoS prevention** | `/audit/logs?limit` clamped to max 500 — previously unbounded |
| **Performance** | `verify_chain()` uses a narrow projection — previously fetched all fields for every entry |
| **Connection leak** | `verify_chain.py` reuses `middleware.get_db()` instead of opening a second `MongoClient` |
| **Web UI** | Interactive dashboard added at `GET /dashboard` — no curl required |

## Constraints

This is an academic prototype. It is not production-hardened:

- No authentication on any API endpoint (`user_id` is caller-supplied)
- No authentication on MongoDB
- Local MongoDB only — no Atlas or cloud setup
- Fake PII only — all records are Faker-generated
- Single machine — no distributed setup
- Fixed dataset — 10 employee records in `sensitive_data`
